Gradle dependency configurations guide

implementation vs api — the most important Gradle distinction
implementation — dependency is an internal implementation detail. Downstream consumers of your library cannot see this dependency's classes at compile time. Use for most dependencies: Spring Boot, Jackson, logging, database drivers. This enables Gradle's compile avoidance — downstream modules are NOT recompiled when this dependency changes internally. api — dependency is exposed in your public API. Use only when your public interfaces return types or take parameters from this dependency. Over-using api defeats compile avoidance and increases build times. When in doubt, use implementation.

Deprecated configurations to avoid
compile — deprecated since Gradle 3.4, removed in Gradle 7. Replace with implementation. testCompile — replace with testImplementation. provided — replace with compileOnly. runtime — replace with runtimeOnly. testRuntime — replace with testRuntimeOnly. Using deprecated configs causes build warnings and will break when upgrading Gradle major versions. Run ./gradlew dependencies to see your resolved dependency tree.

SNAPSHOT dependencies in production builds
SNAPSHOT versions (e.g., 1.0.0-SNAPSHOT) are mutable — the same version string can resolve to different artifacts on different builds. This makes builds non-reproducible. Avoid SNAPSHOT dependencies in production code. Use fixed release versions. In development, pin to a specific SNAPSHOT build number if you need a pre-release. Gradle caches SNAPSHOT dependencies for 24 hours by default — use --refresh-dependencies to force refresh.

FAQ — Analyze Gradle
./gradlew dependencies — full dependency tree with resolved versions. ./gradlew dependencies --configuration runtimeClasspath — specific configuration. ./gradlew dependencyInsight --dependency guava — why a specific artifact is included. ./gradlew build --scan — visual Build Scan in browser (requires Gradle Enterprise). ca.cutterslade.analyze plugin — detects unused and undeclared dependencies. This online tool parses your build.gradle for common issues without requiring Gradle to run.
implementation — internal dependency, not visible to consumers of your library. Enables compile avoidance. Use for ~90% of dependencies. api — exposed in your public API, visible to consumers at compile time. Use only when public methods return or accept types from this dependency. Using api instead of implementation causes unnecessary recompilation of downstream modules — this is a common performance problem in large multi-module builds.
Run ./gradlew dependencies --configuration runtimeClasspath — look for lines showing version conflicts: com.google.guava:guava:20.0 -> 32.1.3-jre (*) means the version was upgraded by Gradle's conflict resolution. (*) means a conflict was resolved. To see details: ./gradlew dependencyInsight --dependency guava. To enforce a version: add to configurations.all { resolutionStrategy { force 'com.google.guava:guava:32.1.3-jre' } }.
Key optimizations: (1) Enable build cache: org.gradle.caching=true in gradle.properties. (2) Enable configuration cache: org.gradle.configuration-cache=true. (3) Enable parallel execution: org.gradle.parallel=true. (4) Use implementation instead of api to enable compile avoidance. (5) Run ./gradlew build --scan to identify slow tasks. (6) Use ./gradlew assembleDebug instead of ./gradlew build for Android. (7) Increase heap: org.gradle.jvmargs=-Xmx4g.
